# Rapport — Deload / Périodisation — Groupe C (Intégration + Streak) — 20260311-2000

## Objectif
Intégrer le moteur de deload (Groupe A) et le composant UI (Groupe B) dans le HomeScreen.
Gérer le cas du streak en semaine de deload. Réconcilier les types si nécessaire.

## Fichiers concernés
- `mobile/src/screens/HomeScreen.tsx` — intégrer `DeloadRecommendationCard`
- `mobile/src/model/utils/gamificationHelpers.ts` — adapter `updateStreak()` pour deload
- `mobile/src/model/models/User.ts` — ajouter champ optionnel `deloadAcknowledgedWeek` (si schema change)
- `mobile/src/model/schema.ts` — v36 : ajouter colonne `deload_acknowledged_week` à `users` (optionnel)

## Contexte technique

### HomeScreen structure actuelle
```
1. Header card (greeting + KPIs)
2. Gamification card (level/XP/streak/badges)
3. Quick-start card (conditional)
4. Weekly activity card (7-day view)        ← INSÉRER ICI
5. Training/Stats tile grids
```

Le `DeloadRecommendationCard` s'insère entre la weekly activity card et les tiles (position 4.5).
Il est **conditionnel** — affiché uniquement quand `computeDeloadRecommendation()` retourne non-null.

### HomeScreen data flow
HomeScreen utilise `withObservables` pour observer :
- `user` (User model)
- `histories` (collection)
- `exercises` (collection)

Les stats sont calculées via les helpers (`computeVolumeStats`, `buildWeeklyActivity`, etc.)
dans un `useMemo` à l'intérieur du composant `HomeContent`.

### Streak logic (`gamificationHelpers.ts`)
```typescript
export async function updateStreak(database: Database, user: User): Promise<void> {
  // Calcule les sessions de la semaine ISO courante
  // Compare avec streakTarget
  // Si sessions >= target → streak++
  // Sinon → streak reset
}
```

**Problème deload :** Si l'utilisateur suit la recommandation de deload week et réduit
ses sessions en dessous de `streakTarget`, son streak est cassé.

**Solution proposée (sans schema change) :** Ne PAS modifier la logique de streak pour le MVP.
Le streak reflète la réalité. On ajoute simplement un message rassurant dans la card deload
quand `currentStreak > 0` : "Ton streak ne sera pas affecté si tu maintiens [target] sessions."

**Solution future (schema v36) :** Ajouter `deload_acknowledged_week` (string, optional) sur `users`.
Quand l'utilisateur clique "Compris" sur une recommendation `deload_week`, on stocke l'ISO week.
`updateStreak()` peut alors accorder un pass pour cette semaine.
→ **Reporter en v2. Ne PAS implémenter le schema change dans ce groupe.**

### Dismiss logic (sans schema change)
Utiliser un `useState` local dans HomeScreen pour tracker le dismiss pendant la session app.
Le recommandation réapparaîtra au prochain lancement — acceptable pour le MVP.

```tsx
const [dismissedDeload, setDismissedDeload] = useState(false)
```

## Étapes

### 1. Préparer les données dans HomeScreen

Dans `HomeContent`, ajouter un `useMemo` qui appelle `computeDeloadRecommendation()` :

```tsx
import { computeDeloadRecommendation } from '../model/utils/statsHelpers'
import { DeloadRecommendationCard } from '../components/DeloadRecommendationCard'

// Dans HomeContent :
const deloadRecommendation = useMemo(() => {
  if (!histories.length) return null

  // Extraire les volumes hebdomadaires depuis les stats déjà calculées
  // (réutiliser volumeStats si déjà calculé, sinon calculer)
  const weeklyVolumes = volumeStats?.weeklyData?.map(w => w.volume) ?? []

  // Extraire les données minimales des histories
  const historyData = histories
    .filter(h => !h.deletedAt && !h.isAbandoned)
    .map(h => ({ startTime: h.startTime, endTime: h.endTime }))

  return computeDeloadRecommendation({
    histories: historyData,
    weeklyVolumes,
    setsPerMuscle: muscleStats?.currentWeek,  // si disponible
    userLevel: user.userLevel ?? 'intermediate',
    currentStreak: user.currentStreak ?? 0,
  })
}, [histories, volumeStats, muscleStats, user])

const [dismissedDeload, setDismissedDeload] = useState(false)
```

### 2. Insérer le composant dans le JSX

Après la weekly activity card, avant les tiles :

```tsx
{deloadRecommendation && !dismissedDeload && (
  <DeloadRecommendationCard
    recommendation={deloadRecommendation}
    onDismiss={() => setDismissedDeload(true)}
  />
)}
```

### 4. Adapter les données volumeStats

Vérifier que `computeVolumeStats()` retourne bien des données exploitables pour `weeklyVolumes`.
Le type retourné par `computeVolumeStats()` dans `statsVolume.ts` inclut :
```typescript
interface VolumeStats {
  weeklyData: Array<{ week: string; volume: number; sets: number }>
  // ... autres champs
}
```

Mapper : `volumeStats.weeklyData.map(w => w.volume)` — mais vérifier l'ordre (récent en premier ou en dernier).
Si l'ordre est chronologique (ancien → récent), inverser : `.reverse()`.

### 5. Ajouter les muscleStats si pas déjà observés

Si `computeSetsPerMuscleWeek()` n'est pas déjà appelé dans HomeScreen :
- Soit l'ajouter dans un `useMemo` (si les données exercices sont déjà chargées)

- **Préférer la simplicité** : passer `undefined` pour le MVP, la surcharge musculaire sera visible dans les stats détaillées
